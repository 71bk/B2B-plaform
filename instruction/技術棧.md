# ContractFlow 技術棧

> B2B 合約與退費審核 SaaS 平台  
> Technical Stack Document

---

## 1. 技術選型原則

ContractFlow 的技術選型以後端工程能力展示為核心，不追求過度複雜，而是優先選擇企業後端常見、可維護、可測試、可部署的技術。

主要原則：

1. 使用成熟穩定的 Java / Spring 生態。
2. 使用 PostgreSQL 作為主要資料庫，支援關聯資料、JSONB、索引與交易一致性。
3. 使用 Spring Security + JWT Cookie 實作 SaaS 後台常見的登入、組織切換與權限控管。
4. 使用 Flyway 管理資料庫版本，讓 schema 可追蹤、可重建。
5. 所有 tenant-owned resource 以 `organization_id` 做資料隔離，避免跨租戶存取。
6. 使用 JUnit、Mockito、Spring Security Test 建立可驗證的後端品質。
7. Redis、Docker、Testcontainers、AI / RAG 放在第二階段後逐步加入，避免 MVP 範圍過大。

---

## 2. MVP 技術棧

MVP 目標是先完成一套可登入、可建立 Organization、可依組織角色審核、可追蹤紀錄的 SaaS 後端系統。

| 類別 | 技術 | 用途 |
|---|---|---|
| 語言 | Java 17 | 後端主要開發語言 |
| 框架 | Spring Boot | API 與應用程式框架 |
| 安全 | Spring Security | 登入驗證、RBAC、Method Security |
| ORM | Spring Data JPA | Entity 與 Repository 資料存取 |
| 驗證 | Jakarta Bean Validation | Request DTO 欄位驗證 |
| 資料庫 | PostgreSQL | 主資料庫 |
| Migration | Flyway | 資料庫版本控管 |
| 認證 | JWT + HttpOnly Cookie | Access Token / Refresh Token |
| 測試 | JUnit 5 | 單元測試 |
| Mock | Mockito | Service 測試 |
| 安全測試 | Spring Security Test | 角色與授權測試 |
| 建置 | Maven | 依賴管理與建置 |

## 3. 第二階段技術棧

第二階段目標是讓系統更接近真實企業產品。

| 類別 | 技術 | 用途 |
|---|---|---|
| 快取 | Redis | Dashboard 統計快取、Refresh Token 儲存 |
| 檔案儲存 | Local Storage / MinIO | 文件上傳、下載與儲存 |
| 整合測試 | Testcontainers | 使用真實 PostgreSQL 測試環境 |
| 容器化 | Docker | 後端服務容器化 |
| 本機環境 | Docker Compose | Backend + PostgreSQL + Redis + MinIO 一鍵啟動 |
| CI/CD | GitHub Actions | 測試、建置、Docker image 流程 |
| 反向代理 | Nginx | 部署時代理前後端服務 |
| 監控 | Spring Boot Actuator | Health、metrics、info |

## 4. 前端技術棧

前端定位為 SaaS 後台工作台，介面應偏向清楚、穩定、可快速操作。

| 類別 | 技術 | 用途 |
|---|---|---|
| 框架 | Next.js | React App Router、建置與部署 |
| UI Library | React | 後台管理介面 |
| 語言 | TypeScript | 型別安全與大型前端維護 |
| 樣式 | Tailwind CSS | Utility-first 樣式系統 |
| UI 元件 | shadcn/ui | 表格、表單、彈窗、按鈕等後台元件基礎 |
| Server State | TanStack Query | API 查詢、快取、loading/error 狀態 |
| 表單 | React Hook Form | 表單狀態與提交流程 |
| 驗證 | Zod | 表單與 API schema 驗證 |
| HTTP | Fetch / Axios | 呼叫後端 API |

前端不是 MVP 第一優先，後端 API 穩定後再開始實作。ContractFlow 是 SaaS 後台，不是公開 SEO 網站，因此 MVP 主要以 Client Components、Organization switcher 與 API 串接為主，不需要一開始深入 SSR。

## 5. AI / RAG 技術棧

AI 功能屬於作品集加分項，不進入 MVP。

| 類別 | 技術 | 用途 |
|---|---|---|
| Worker | Python FastAPI | AI Worker API |
| 向量資料庫 | PostgreSQL pgvector | 合約 chunk embedding 儲存 |
| 文件解析 | PDF Parser / OCR | 合約文字抽取 |
| Embedding | Embedding Model | 文件向量化 |
| RAG | Retrieval Augmented Generation | 合約搜尋、案件摘要、缺件提醒 |

AI 不負責自動核准或拒絕案件，只提供搜尋、摘要與提醒。

## 6. 後端分層與模組

後端採用清楚的分層架構：

```text
Controller Layer
    ↓
Application Service Layer
    ↓
Domain / Business Logic Layer
    ↓
Repository Layer
    ↓
Database
```

建議模組：

| 模組 | 說明 |
|---|---|
| common | 統一回應、例外處理、traceId、工具類 |
| auth | Login、Logout、Refresh Token、目前使用者 |
| organizations | Organization、成員、邀請、租戶 context |
| users | 使用者、角色、權限 |
| clients | B2B 客戶資料 |
| projects | 專案與合約資料 |
| refunds | 退費案件、狀態機、審核流程 |
| audit | 操作紀錄與狀態紀錄 |
| files | 文件上傳與下載 |
| dashboard | 統計 API |
| notifications | 站內通知與 Email 通知 |
| subscriptions | 訂閱方案、組織方案狀態與用量限制 |
| ai | AI 搜尋與摘要 API |

## 7. 資料庫規範

### 7.1 型別規範

| 資料 | PostgreSQL | Java |
|---|---|---|
| 主鍵 | BIGSERIAL / BIGINT | Long |
| 金額 | NUMERIC(12,2) | BigDecimal |
| 狀態 | VARCHAR(20-50) | Enum |
| 時間 | TIMESTAMP | LocalDateTime |
| 日期 | DATE | LocalDate |
| JSON | JSONB | String / JsonNode |
| 是否 | BOOLEAN | Boolean |

### 7.2 一致性規範

1. 金額欄位一律使用 `NUMERIC(12,2)` 與 `BigDecimal`。
2. 多人可能同時修改的資料表需加入 `version` 欄位，使用 JPA optimistic locking。
3. 所有主要資料表需有 `created_at` 與 `updated_at`。
4. 重要狀態變更需同時寫入 status log 與 audit log。
5. FK、unique constraint、index 由 Flyway migration 明確管理。
6. `clients`、`projects`、`contracts`、`refund_cases`、`files`、`notifications`、`operation_audit_logs` 必須帶 `organization_id`。

### 7.3 查詢索引

優先建立以下索引：

| 資料表 | 索引用途 |
|---|---|
| users.email | 登入查詢 |
| organization_members.organization_id + user_id | 驗證使用者是否屬於組織 |
| clients.organization_id | 查詢組織客戶 |
| projects.organization_id | 查詢組織專案 |
| contracts.organization_id | 查詢組織合約 |
| refund_cases.organization_id + status | 組織案件狀態篩選 |
| refund_cases.organization_id + applicant_id | Sales 查詢自己的案件 |
| refund_cases.organization_id + current_assignee_id | 待處理案件查詢 |
| refund_cases.organization_id + created_at | 案件時間排序 |
| operation_audit_logs.organization_id + resource_type + resource_id | 查看資源操作歷史 |
| refund_case_status_logs.refund_case_id | 查看案件狀態歷史 |

## 8. 安全技術規範

### 8.1 認證

系統採用 JWT + HttpOnly Cookie：

- Access Token：短效，用於 API 驗證。
- Refresh Token：長效，用於更新 Access Token。
- Cookie：`HttpOnly`、`Secure`、`SameSite=Lax` 或 `SameSite=Strict`。
- 若前後端跨站且必須使用 `SameSite=None`，需啟用 CSRF Token。

### 8.2 授權

授權採用 RBAC：

```text
User → OrganizationMember → Role → Permission
```

實作原則：

1. Controller 使用 `@PreAuthorize` 做角色或權限檢查。
2. Service 層先檢查使用者是否屬於目前 Organization，再做資源層級檢查。
3. Sales 只能操作同一 Organization 內自己建立的草稿案件。
4. 狀態轉換必須檢查目前狀態、操作角色與目標狀態。
5. System Admin 具備平台管理權限，但跨組織支援操作仍需寫入 Audit Log。

### 8.3 CSRF

因為系統使用 Cookie 驗證，寫入型 API 需考慮 CSRF：

- 同站部署：優先使用 `SameSite=Lax` 或 `SameSite=Strict`。
- 跨站部署：使用 CSRF Token，例如 Double Submit Cookie 或 `X-CSRF-TOKEN` Header。
- CORS 僅允許指定前端網域，不允許任意 origin。

## 9. 測試技術規範

MVP 至少要覆蓋以下測試：

| 測試類型 | 技術 | 重點 |
|---|---|---|
| Unit Test | JUnit 5 | 狀態轉換、金額驗證、Service 邏輯 |
| Mock Test | Mockito | Repository / 外部服務隔離 |
| Security Test | Spring Security Test | 不同角色 API 權限 |
| Integration Test | Spring Boot Test | API + DB 整合 |
| DB Test | Testcontainers | 第二階段加入真實 PostgreSQL 測試 |

必測方向：

1. Sales 可以建立案件，但不能核准案件。
2. Admin 可以退回或通過內勤審核。
3. Finance 可以確認退款金額與標記已退款。
4. Manager 可以核准或拒絕案件。
5. 狀態不可跳關。
6. 已退款案件不可再次修改。
7. 同一案件被兩個人同時審核時，後提交者會因版本衝突失敗。
8. 重要操作必須寫入 Audit Log。
9. 使用者不可跨 Organization 存取資料。

## 10. 開發優先順序

```text
1. Spring Boot 專案骨架
2. PostgreSQL + Flyway
3. 統一 API Response
4. Global Exception Handler
5. User / Role / Permission
6. Organization / Member / Invitation
7. Login / Logout / Refresh Token
8. Spring Security + Organization-aware RBAC
9. Client / Project / Contract
10. Refund Case + 狀態機
11. Status Log + Audit Log
12. 測試
13. File Upload
14. Dashboard
15. Docker Compose
16. Redis
17. AI / RAG
```

## 11. 不建議在 MVP 提早導入的技術

以下技術不是不能用，而是會增加第一版複雜度，建議核心流程穩定後再導入：

| 技術 | 延後原因 |
|---|---|
| Redis | 核心流程不依賴快取，Dashboard 完成後再加 |
| MinIO | 文件上傳可先用 Local Storage 驗證流程 |
| Testcontainers | 初期可先用單元測試與 Spring Boot Test，之後再加真實 DB 測試 |
| AI / RAG | 屬於作品集加分項，不應阻塞審核流程 MVP |
| Message Queue | 通知量不大時可先同步建立站內通知 |
