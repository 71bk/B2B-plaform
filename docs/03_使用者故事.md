# ContractFlow 使用者故事

> B2B 合約與退費審核 SaaS 平台  
> User Stories Document

---

## 1. 角色定義

| 角色 | 說明 |
|---|---|
| Owner | 組織擁有者，負責組織設定、成員邀請與角色分派 |
| Sales | 業務人員，負責建立退費案件、上傳資料、查看自己的案件 |
| Admin | 內勤人員，負責確認案件資料完整性與退回補件 |
| Finance | 財務人員，負責確認退款金額、付款資訊與標記退款完成 |
| Manager | 主管，負責最終核准或拒絕退費案件 |
| System Admin | SaaS 平台管理員，負責平台維運、方案設定與稽核支援 |

---

## 2. MVP 範圍

MVP 優先完成以下使用者故事：

1. 使用者登入、登出與查看自己身份。
2. 使用者可建立或切換 Organization。
3. Owner 可邀請成員並設定組織內角色。
4. 依 Organization 與角色限制可查看與可操作的資料。
5. Sales 建立客戶、專案、合約與退費案件。
6. 退費案件依照固定流程審核。
7. 每次狀態變更都留下狀態紀錄。
8. 重要操作都留下 Audit Log。
9. 使用者可以查詢與篩選案件。

文件上傳、Dashboard、通知、Redis、AI / RAG 可放到第二階段。

## 3. Epic 1：登入與權限

### US-001 使用者登入

作為組織成員，我希望可以使用 Email 和密碼登入系統，讓我可以進入後台處理案件。

驗收條件：

1. 使用者輸入正確 Email 與密碼時，系統回傳登入成功。
2. 系統透過 HttpOnly Cookie 設定 Access Token。
3. 登入失敗時，系統回傳清楚的錯誤訊息。
4. 登入成功與失敗都需寫入 `login_audit_logs`。

### US-002 使用者登出

作為已登入使用者，我希望可以登出系統，避免其他人使用我的登入狀態。

驗收條件：

1. 使用者登出後，Access Token Cookie 失效。
2. Refresh Token 被撤銷或標記失效。
3. 登出後再次呼叫受保護 API 會被拒絕。

### US-003 查看目前登入者

作為已登入使用者，我希望可以查看自己的基本資料、可用 Organization 與各組織角色，讓前端可以顯示正確功能。

驗收條件：

1. `GET /api/auth/me` 回傳 user id、name、email、organizations 與目前 organization roles。
2. 未登入使用者呼叫時回傳 401。
3. 停用帳號不可取得目前登入資訊。

### US-004 角色權限限制

作為系統，我希望不同角色只能操作被授權的功能，避免越權存取。

驗收條件：

1. Sales 不能執行 Admin / Finance / Manager 的審核 API。
2. Admin 不能執行 Finance 與 Manager 的審核 API。
3. Finance 不能執行 Manager 核准 API。
4. 未登入使用者不可存取受保護 API。
5. 權限不足時回傳 403。
6. 不屬於該 Organization 的使用者不可存取該組織資料。

## 3.5 Epic 1.5：組織與成員管理

### US-ORG-001 建立 Organization

作為新客戶，我希望註冊後可以建立自己的 Organization，讓公司資料與其他客戶隔離。

驗收條件：

1. 建立 Organization 時需輸入組織名稱與唯一 slug。
2. 建立者自動成為該 Organization 的 Owner。
3. 系統建立預設方案狀態。
4. 建立成功後寫入 Audit Log。

### US-ORG-002 切換 Organization

作為加入多個 Organization 的使用者，我希望可以切換目前操作的 Organization，避免在錯誤公司底下建立資料。

驗收條件：

1. 前端可列出使用者可存取的 Organization。
2. 使用者只能切換到自己已加入且狀態有效的 Organization。
3. 切換後所有 API 使用 `/api/orgs/{orgId}/...` 作為租戶 context。

### US-ORG-003 邀請組織成員

作為 Owner，我希望可以用 Email 邀請同事加入 Organization，並指定角色。

驗收條件：

1. Owner 可輸入 Email 並指定 Sales / Admin / Finance / Manager / Owner。
2. 邀請 token 需有到期時間。
3. 被邀請者接受後建立 `organization_members`。
4. 邀請、接受與取消都需寫入 Audit Log。

### US-ORG-004 調整組織成員角色

作為 Owner，我希望可以調整成員在 Organization 內的角色，讓權限符合職責。

驗收條件：

1. 只有 Owner 或 System Admin 可調整角色。
2. 同一使用者在不同 Organization 可有不同角色。
3. 角色變更後立即影響後續 API 授權。
4. 角色變更需寫入 Audit Log。

## 4. Epic 2：客戶、專案與合約管理

### US-005 建立客戶

作為 Sales 或 Admin，我希望可以建立 B2B 客戶資料，讓退費案件可以關聯正確客戶。

驗收條件：

1. 可輸入公司名稱、統一編號、聯絡人、Email、電話。
2. 公司名稱為必填。
3. 統一編號在同一 Organization 內不可重複。
4. 建立成功後寫入 Audit Log。

### US-006 查詢客戶

作為組織成員，我希望可以查詢同一 Organization 的客戶資料，讓我可以建立或審核案件。

驗收條件：

1. 可依公司名稱或統一編號搜尋。
2. 回傳結果需支援分頁。
3. 未登入使用者不可查詢。
4. 不可查詢其他 Organization 的客戶。

### US-007 建立專案

作為 Sales 或 Admin，我希望可以在客戶底下建立專案，讓合約與退費案件可以關聯到具體合作項目。

驗收條件：

1. 專案必須關聯一個既有客戶。
2. 專案與客戶必須屬於同一 Organization。
3. 可輸入專案名稱、合約金額、起訖日期、狀態。
4. 合約金額使用 `BigDecimal`，不可為負數。
5. 建立成功後寫入 Audit Log。

### US-008 建立合約

作為 Sales 或 Admin，我希望可以建立合約資料，讓退費審核時可以確認原合約資訊。

驗收條件：

1. 合約必須關聯一個既有專案。
2. 合約與專案必須屬於同一 Organization。
3. 合約編號在同一 Organization 內不可重複。
4. 可輸入合約名稱、合約金額、簽約日期、合約起訖日期。
5. 合約金額使用 `BigDecimal`，不可為負數。

## 5. Epic 3：退費案件建立與查詢

### US-009 建立退費草稿

作為 Sales，我希望可以建立退費案件草稿，讓我可以先保存資料再送出審核。

驗收條件：

1. 案件建立後狀態為 `DRAFT`。
2. 案件必須關聯同一 Organization 的客戶、專案與申請人。
3. 退費金額不可為負數。
4. 案件編號在同一 Organization 內不可重複。
5. 建立成功後寫入 Audit Log。

### US-010 編輯草稿案件

作為 Sales，我希望可以編輯自己建立的草稿案件，讓資料送出前可以修正。

驗收條件：

1. Sales 只能編輯自己建立的 `DRAFT` 案件。
2. 非 `DRAFT` 案件不可被 Sales 直接修改。
3. 每次修改需更新 `updated_at`。
4. 重要欄位修改需寫入 Audit Log。

### US-011 送出退費案件

作為 Sales，我希望可以送出草稿案件，讓內勤開始審核。

驗收條件：

1. 只有 `DRAFT` 案件可以送出。
2. 送出後狀態改為 `ADMIN_REVIEW`。
3. 系統設定 `submitted_at`。
4. 系統寫入 `refund_case_status_logs`。
5. 系統寫入 `operation_audit_logs`。

### US-012 取消草稿案件

作為 Sales，我希望可以取消尚未送出的草稿案件，避免錯誤案件進入審核流程。

驗收條件：

1. 只有 `DRAFT` 案件可以取消。
2. 只有案件申請人可以取消自己的草稿案件。
3. 取消後狀態改為 `CANCELLED`。
4. 系統寫入 Status Log 與 Audit Log。

### US-013 查詢我的案件

作為 Sales，我希望可以查看自己建立的案件，了解目前處理進度。

驗收條件：

1. Sales 只能看到自己建立的案件。
2. 可依狀態、客戶、建立時間篩選。
3. 查詢結果需支援分頁與排序。

### US-014 查詢全部案件

作為 Admin、Finance 或 Manager，我希望可以查看全部案件，讓我可以處理自己負責的審核工作。

驗收條件：

1. Admin / Finance / Manager 可以查看同一 Organization 的全部案件。
2. 可依狀態、申請人、客戶、日期篩選。
3. 未授權角色不可查看全部案件。

## 6. Epic 4：退費審核流程

### US-015 內勤審核通過

作為 Admin，我希望可以通過內勤審核，讓案件進入財務確認階段。

驗收條件：

1. 只有 `ADMIN_REVIEW` 案件可以內勤通過。
2. 只有 Admin 或 System Admin 可以執行。
3. 通過後狀態改為 `FINANCE_REVIEW`。
4. 系統寫入 Status Log 與 Audit Log。

### US-016 內勤退回補件

作為 Admin，我希望可以退回資料不完整的案件，讓 Sales 補正後重新送出。

驗收條件：

1. 只有 `ADMIN_REVIEW` 案件可以退回。
2. 退回時必須填寫原因。
3. 退回後狀態改為 `DRAFT`。
4. 系統寫入 Status Log 與 Audit Log。

### US-017 財務確認退款

作為 Finance，我希望可以確認退款金額與付款資訊，讓案件進入主管核准。

驗收條件：

1. 只有 `FINANCE_REVIEW` 案件可以財務確認。
2. 只有 Finance 或 System Admin 可以執行。
3. 財務可確認或修正退款金額。
4. 確認後狀態改為 `MANAGER_REVIEW`。
5. 系統寫入 Status Log 與 Audit Log。

### US-018 財務退回補件

作為 Finance，我希望可以退回付款資訊不完整的案件，讓 Sales 補正資料。

驗收條件：

1. 只有 `FINANCE_REVIEW` 案件可以退回。
2. 退回時必須填寫原因。
3. 退回後狀態改為 `DRAFT`。
4. 系統寫入 Status Log 與 Audit Log。

### US-019 主管核准案件

作為 Manager，我希望可以核准退費案件，讓財務可以安排退款。

驗收條件：

1. 只有 `MANAGER_REVIEW` 案件可以核准。
2. 只有 Manager 或 System Admin 可以執行。
3. 核准後狀態改為 `APPROVED`。
4. 系統設定 `approved_at`。
5. 系統寫入 Status Log 與 Audit Log。

### US-020 主管拒絕案件

作為 Manager，我希望可以拒絕不符合規定的退費案件，讓案件流程結束。

驗收條件：

1. 只有 `MANAGER_REVIEW` 案件可以拒絕。
2. 拒絕時必須填寫原因。
3. 拒絕後狀態改為 `REJECTED`。
4. 被拒絕案件不可再次審核。
5. 系統寫入 Status Log 與 Audit Log。

### US-021 財務標記已退款

作為 Finance，我希望可以把已完成付款的案件標記為已退款，讓案件流程正式結束。

驗收條件：

1. 只有 `APPROVED` 案件可以標記已退款。
2. 只有 Finance 或 System Admin 可以執行。
3. 標記後狀態改為 `PAID`。
4. 系統設定 `paid_at`。
5. `PAID` 案件不可再次修改。
6. 系統寫入 Status Log 與 Audit Log。

## 7. Epic 5：紀錄與稽核

### US-022 查看案件狀態紀錄

作為組織成員，我希望可以查看同一 Organization 案件的狀態變更歷史，了解案件經過哪些處理。

驗收條件：

1. 每次狀態變更都會新增一筆 `refund_case_status_logs`。
2. 紀錄包含 from status、to status、action、actor、comment、created time。
3. 有案件查看權限的使用者才能查看該案件狀態紀錄。

### US-023 寫入操作紀錄

作為系統，我希望重要操作都寫入 Audit Log，讓每個 Organization 可以追蹤誰在什麼時間做了什麼事。

驗收條件：

1. 建立、修改、審核、退回、拒絕、標記已退款都需寫入 Audit Log。
2. Audit Log 包含 organization、actor、action、resource type、resource id、before data、after data、ip、user agent、trace id。
3. Audit Log 不允許由一般 API 修改或刪除。

## 8. Epic 6：併發與資料一致性

### US-024 防止同時審核覆蓋

作為系統，我希望同一案件被多人同時操作時不會互相覆蓋，避免錯誤狀態產生。

驗收條件：

1. `refund_cases` 使用 `version` 欄位做 optimistic locking。
2. 兩個人同時送出審核操作時，後提交者會收到版本衝突錯誤。
3. 版本衝突不應寫入錯誤的狀態紀錄。

### US-025 防止狀態跳關

作為系統，我希望案件只能依照定義好的流程前進，避免非法狀態轉換。

驗收條件：

1. `DRAFT` 不可直接變成 `FINANCE_REVIEW`。
2. `ADMIN_REVIEW` 不可直接變成 `APPROVED`。
3. `PAID`、`REJECTED`、`CANCELLED` 為終止狀態，不可再被修改。
4. 非法狀態轉換回傳明確錯誤訊息。

## 9. 第二階段使用者故事

### US-026 上傳案件文件

作為 Sales，我希望可以上傳合約或退款相關文件，讓審核人員可以集中查看資料。

驗收條件：

1. 可上傳指定格式文件。
2. 文件需關聯退費案件。
3. 使用者只能上傳自己有權限操作的案件文件。
4. 上傳成功後寫入 Audit Log。

### US-027 查看 Dashboard

作為 Admin、Finance 或 Manager，我希望可以查看退費統計，了解目前案件量、金額與處理效率。

驗收條件：

1. 可查看不同狀態案件數量。
2. 可查看月度退費金額趨勢。
3. 可查看退費原因排名。
4. Dashboard 查詢可使用 Redis 快取。

### US-028 接收站內通知

作為組織成員，我希望案件輪到我處理時收到通知，避免案件卡關。

驗收條件：

1. 案件送出後通知 Admin。
2. 內勤通過後通知 Finance。
3. 財務確認後通知 Manager。
4. 主管核准後通知 Finance。
5. 案件被退回或拒絕時通知 Sales。

### US-029 AI 合約搜尋

作為審核人員，我希望可以搜尋合約條款，快速確認退費相關內容。

驗收條件：

1. 可輸入問題查詢合約內容。
2. 系統回傳答案與引用來源。
3. AI 僅作輔助，不自動決定案件是否核准。

### US-030 AI 案件摘要

作為 Manager，我希望可以快速產生案件摘要，協助我理解案件背景與處理進度。

驗收條件：

1. 系統根據案件資料、狀態紀錄、留言與文件摘要產生內容。
2. 摘要需標示資料來源。
3. 使用者仍需自行做最終決策。

## 10. 非功能性使用者故事

### NFR-001 安全性

作為 SaaS 平台，我需要保護使用者身份、租戶資料、案件資料與合約文件。

驗收條件：

1. 密碼使用 BCrypt 或 Argon2id Hash。
2. Cookie 設定 HttpOnly、Secure、SameSite。
3. 寫入型 API 需考慮 CSRF 防護。
4. CORS 只允許指定前端網域。
5. 不同 Organization 的資料必須隔離。

### NFR-002 可追蹤性

作為系統維護者，我需要能追蹤每筆 API 與重要操作。

驗收條件：

1. 每筆 API Request 具備 traceId。
2. 錯誤回應包含 traceId。
3. 重要操作寫入 Audit Log。

### NFR-003 可測試性

作為開發者，我需要能用測試驗證核心流程，避免重構後破壞審核規則。

驗收條件：

1. 狀態轉換邏輯有 Unit Test。
2. 角色權限有 Security Test。
3. 主要 API 有 Integration Test。
4. 測試需覆蓋不可跳關、不可越權與終止狀態不可修改。

## 11. MVP 完成標準

MVP 完成時，系統至少能展示以下完整流程：

```text
Sales 登入
    ↓
選擇或建立 Organization
    ↓
建立客戶、專案與合約
    ↓
建立退費草稿
    ↓
送出案件
    ↓
Admin 內勤審核
    ↓
Finance 財務確認
    ↓
Manager 主管核准
    ↓
Finance 標記已退款
    ↓
查看案件狀態紀錄與 Audit Log
```

完成這條流程後，再進入文件上傳、Dashboard、通知、Redis、Docker 與 AI 功能。
