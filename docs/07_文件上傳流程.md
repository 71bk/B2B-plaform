# 文件上傳流程

> Contract / Refund Document Upload Flow

---

## 功能定位

文件上傳屬於第二階段功能，但資料表與 API 在設計時先保留擴充空間。

文件用途：

1. 合約文件。
2. 退款申請附件。
3. 退款帳戶證明。
4. 主管或財務需要的補充文件。
5. 退款完成證明。

---

## 檔案分類

| file_type | 說明 |
|---|---|
| CONTRACT | 合約文件 |
| REFUND_APPLICATION | 退費申請文件 |
| BANK_ACCOUNT_PROOF | 退款帳戶證明 |
| REFUND_PROOF | 退款完成證明 |
| OTHER | 其他附件 |

---

## 上傳流程

```text
使用者選擇檔案
    ↓
前端呼叫上傳 API
    ↓
後端檢查登入、Organization membership 與案件權限
    ↓
驗證副檔名、MIME type、檔案大小
    ↓
儲存檔案到 Local Storage / MinIO
    ↓
寫入 files
    ↓
寫入 refund_case_files
    ↓
寫入 Audit Log
```

---

## MVP 與第二階段策略

MVP 可先不實作文件上傳，只保留 API 與資料表設計。

第二階段建議：

1. 開發環境先使用 Local Storage。
2. Docker Compose 加入 MinIO。
3. Production 可換成 S3-compatible storage。
4. 下載檔案時一律經過後端權限檢查。

---

## 權限規則

| 角色 | 上傳 | 下載 | 刪除 |
|---|---|---|---|
| Owner | 同一 Organization 案件 | 同一 Organization 案件 | 同一 Organization 文件 |
| Sales | 同一 Organization 內自己的案件 | 同一 Organization 內自己的案件 | 自己上傳且案件未結束 |
| Admin | 同一 Organization 可處理案件 | 同一 Organization 可處理案件 | 補件相關文件 |
| Finance | 同一 Organization 財務審核案件 | 同一 Organization 財務審核案件 | 退款證明 |
| Manager | 同一 Organization 審核案件 | 同一 Organization 審核案件 | 不建議開放 |
| System Admin | 平台支援操作 | 平台支援操作 | 平台支援操作 |

---

## 安全限制

1. 不信任前端傳入的 content type，後端需重新判斷。
2. 原始檔名不可直接作為儲存路徑。
3. 儲存路徑需使用 UUID 或 hash。
4. 限制檔案大小，例如單檔 10MB。
5. 僅允許 PDF、PNG、JPG、DOCX 等白名單格式。
6. 下載前需檢查使用者是否有案件查看權限。
7. `files` 必須帶 `organization_id`，下載與刪除不可跨 Organization。

---

## Audit Log

以下操作需寫入 `operation_audit_logs`：

1. 上傳文件。
2. 下載敏感文件。
3. 刪除文件。
4. 合約文件被替換。
