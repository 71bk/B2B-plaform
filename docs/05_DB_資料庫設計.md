# 資料庫設計

> ContractFlow PostgreSQL Schema Design

---

## 命名規範

| 項目 | 規範 |
|---|---|
| schema | MVP 使用 `public`，第二階段可拆 `app` / `audit` / `vector` |
| table | snake_case 複數，例如 `refund_cases` |
| column | snake_case |
| PK | `id BIGSERIAL` |
| FK | `{resource}_id BIGINT` |
| timestamp | `created_at`、`updated_at` |
| soft delete | 使用 `deleted_at` |
| optimistic lock | 使用 `version BIGINT` |

## 使用者與權限

### users

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | BIGSERIAL | PK |
| email | VARCHAR(255) | unique |
| password_hash | VARCHAR(255) | BCrypt / Argon2id |
| name | VARCHAR(100) | 姓名 |
| status | VARCHAR(20) | ACTIVE / DISABLED |
| version | BIGINT | optimistic locking |
| created_at | TIMESTAMP | 建立時間 |
| updated_at | TIMESTAMP | 更新時間 |

### roles / permissions

`roles.code` 與 `permissions.code` 需唯一。角色透過 `role_permissions` 對應權限。SaaS 租戶內角色以 `organization_members.role_id` 為主，`user_roles` 僅保留給平台層級角色或相容既有 RBAC 設計。

### refresh_tokens

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | BIGSERIAL | PK |
| user_id | BIGINT | FK users.id |
| token_hash | VARCHAR(255) | Refresh Token hash |
| revoked | BOOLEAN | 是否撤銷 |
| expires_at | TIMESTAMP | 到期時間 |
| created_at | TIMESTAMP | 建立時間 |

### login_audit_logs

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | BIGSERIAL | PK |
| user_id | BIGINT | FK users.id，可為空 |
| email | VARCHAR(255) | 嘗試登入的 Email |
| success | BOOLEAN | 是否成功 |
| failure_reason | VARCHAR(255) | 失敗原因 |
| ip_address | VARCHAR(45) | IP |
| user_agent | TEXT | User Agent |
| created_at | TIMESTAMP | 建立時間 |

## Migration 規則

1. 使用 Flyway，檔名格式：`V{number}__{description}.sql`。
2. 已套用的 migration 不可修改，只能新增下一版。
3. DDL 需包含 constraint 與 index，不只建 table。
4. Seed roles / permissions 可放在 migration 或 bootstrap runner，但需可重建。
5. schema 變更需同步更新本文件。
6. 所有 tenant-owned table 新增時必須包含 `organization_id`、FK 與查詢索引。
