# 後端開發規範

---

## 分層與命名

後端採用：

```text
Controller
Service
Domain
Repository
Entity
DTO
Mapper
```

規則：

1. Controller 只處理 HTTP、request validation、response。
2. Service 處理交易邊界與 application flow。
3. Domain 放狀態轉換與核心業務規則。
4. Repository 只處理資料存取。
5. Entity 不直接暴露給 API response。

## DTO / VO

1. Request DTO 使用 Bean Validation。
2. Response DTO 不回傳 password hash、token、內部欄位。
3. 分頁 response 統一使用 PageResponse。
4. 金額使用 BigDecimal。
5. 狀態使用 enum，不使用任意字串流動。

---

## Exception 與 ErrorCode

使用自訂 exception：

| Exception | 場景 |
|---|---|
| `BusinessException` | 業務錯誤 |
| `ResourceNotFoundException` | 查無資料 |
| `ForbiddenException` | 權限不足 |
| `InvalidStateTransitionException` | 狀態不可轉換 |
| `VersionConflictException` | optimistic locking 衝突 |

所有錯誤由 Global Exception Handler 統一轉換。

---

## Transaction

1. 寫入操作需標註 `@Transactional`。
2. 查詢操作可使用 `@Transactional(readOnly = true)`。
3. 狀態轉換、status log、audit log 必須在同一交易中。
4. 呼叫外部服務不要放在長交易內。

---

## Security

1. Controller 使用 `@PreAuthorize`。
2. Service 層必須做 Organization membership 與資源擁有者檢查。
3. 不信任前端傳入的 userId。
4. 不信任前端傳入的 `orgId`，必須驗證目前使用者屬於該 Organization。
5. 目前使用者從 SecurityContext 取得。
6. 審核操作必須檢查角色、狀態與 `organization_id`。

---

## Multi-tenancy

1. Tenant-owned API 使用 `/api/orgs/{orgId}/...`。
2. `clients`、`projects`、`contracts`、`refund_cases`、`files`、`notifications`、`operation_audit_logs` 查詢必須帶 `organization_id`。
3. Repository method 命名需明確包含 Organization，例如 `findByOrganizationIdAndId`。
4. 建立 tenant-owned resource 時，`organization_id` 由後端從 request path 與 membership 驗證後設定，不由前端 body 決定。
5. Audit Log 必須記錄 `organization_id`；平台級操作可為空。

---

## Code Style

1. 類別名稱清楚表達用途。
2. 不在 Controller 寫商業邏輯。
3. 不使用 `double` 表示金額。
4. 不吞 exception。
5. 不把 Entity 直接序列化成 API response。
6. 不在 migration 外手動改 schema。
7. 不寫未帶 `organization_id` 的 tenant-owned resource 查詢。
